# Viewing Without pdf.js

This page is the second viewer implementation in the repo, and it is intentionally separate from the
`pdf.js` demo.

It does **not** import:

- `pdfjs-dist`
- a `pdf.js` worker
- Mozilla's viewer shell
- `TextLayerBuilder`
- `AnnotationLayerBuilder`

Instead, it uses **LibPDF only** to parse a document, build a LibPDF-native render plan for common
paint operators, and layer extracted text plus annotations on top.

<Callout type="info" title="What This Demo Now Renders">
  The standalone LibPDF path now paints vector paths, image XObjects, gradients, clipping, and Form
  XObjects on canvas, then adds a searchable text overlay and annotation regions on top. Unsupported
  operators are reported per document as runtime render warnings instead of being hidden behind a
  generic limitation banner.
</Callout>

## Architecture

The no-`pdf.js` pipeline in this repo is:

1. Browser uploads the PDF or loads the sample file
2. Next.js route parses the bytes with LibPDF
3. LibPDF builds a render plan from page content streams, XObjects, graphics state, and gradients
4. The browser executes that render plan on canvas
5. The browser overlays extracted text, search highlights, and annotations

That means this implementation now has a real display layer instead of only an HTML reconstruction
layer.

## Why this exists

You asked for a completely separate viewer built from scratch, without `pdf.js` anywhere in the
path. This page is that implementation.

It is useful for:

- validating that LibPDF has enough parsing APIs to support viewing features
- proving that a custom viewer can exist without Mozilla's rendering stack
- surfacing the exact remaining edge cases as concrete render warnings
